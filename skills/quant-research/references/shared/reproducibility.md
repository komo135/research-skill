# reproducibility.md

The reproducibility 3-tuple — data hash + git commit + env lock — is the
minimum rerun anchor for a trial. It identifies the inputs, analysis code,
and dependency lock used for a result. Required for every promotion-eligible
or claim-cited trial in both R&D and Pure Research.

## When to read

- Setting up a new project's `reproducibility/` folder
- About to run a promotion-eligible or claim-cited trial (must stamp the 3-tuple)
- Reviewing a promotion claim (verify 3-tuple recorded)
- Operating `scripts/reproducibility_stamp.py`

## The principle

A claim that cannot be traced to its inputs, code, and dependency lock is not
defensible. The 3-tuple is the cheapest way to make reruns possible:

- **Data hash**: anchors the input data
- **Git commit**: anchors the analysis code
- **Env lock**: anchors the dependency versions

Together, these three identifiers anchor the computational environment of the
trial. They do not prove cross-hardware identity: OS, system libraries, CPU /
GPU differences, external services, and nondeterministic training can still
matter. For stochastic algorithms, seeds and repeated runs remain necessary.

## Reproducibility levels

Use the lightest level that matches the claim:

| Level | Meaning | Typical evidence |
|---|---|---|
| Traceable | A run can be understood later | run note, parameters, metric, artifact link |
| Rerunnable | A run has stable rerun anchors | data hash, git commit, env lock, seed |
| Reproduced | The result was rerun and matched within tolerance | verification output, tolerance, rerun timestamp |
| Validated | An independent reviewer or environment confirmed the result | review note or external validation artifact |

Exploratory, smoke-test, and debugging runs may stop at Traceable. Promotion
and external claims require Rerunnable at minimum; Reproduced or Validated is
preferred when the claim is high-impact.

## The 3-tuple components

### 1. Data hash

A SHA-256 of every data file used in the trial. Recorded in
`reproducibility/data_hashes.txt`:

```
# format: <relative path>  <sha256>
data/raw/spx_5min_2010_2024.parquet  a3f7d2e1b9c4f5a2e1d8b7c6a5f4e3d2c1b0a9f8e7d6c5b4a3f2e1d0c9b8a7f6
data/raw/vix_daily_1990_2024.parquet  b9c4f5a2e1d8b7c6a5f4e3d2c1b0a9f8e7d6c5b4a3f2e1d0c9b8a7f6e5d4c3b2
literature/papers_index.json  c5b4a3f2e1d0c9b8a7f6e5d4c3b2a1f0e9d8c7b6a5f4e3d2c1b0a9f8e7d6c5b4
```

Generated by `scripts/reproducibility_stamp.py` which scans the data
sources used by the trial and records their hashes.

**Convention**: the hash covers the **content**, not the file
metadata. Renaming the file or moving it does not change the hash.

**When data changes** (vendor update, period extension, etc.): file
a deviation entry in `decisions.md`, recompute the hash, update
`data_hashes.txt`, and verify whether downstream trials need to be
re-run with the new hash. This is non-trivial — data updates are a
common silent source of irreproducibility.

### 2. Git commit hash

The Git commit hash of the analysis code at the moment the trial ran.
Recorded automatically by `scripts/reproducibility_stamp.py`:

```bash
git rev-parse HEAD
# → 1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b
```

**Convention**: the commit must be on a branch (not a detached HEAD)
and the working tree must be clean. If the working tree has uncommitted
changes when the trial runs, `reproducibility_stamp.py` returns a
non-zero exit code (the trial cannot be considered reproducible until
changes are committed).

**Best practice**: use a separate Git branch per project (`project_<name>/main`)
so the commit history is isolated from other projects' work.

### 3. Env lock

A SHA-256 of the `uv.lock` file (or equivalent) that pins the Python
environment used at trial time.

```
reproducibility/uv.lock  <full lock file at trial time, SHA-256 in env_lock_hash.txt>
reproducibility/env_lock_hash.txt  → 7d6c5b4a3f2e1d0c9b8a7f6e5d4c3b2a1f0e9d8c7b6a5f4e3d2c1b0a9f8e7d6c
```

**Tool choice**: this skill standardizes on `uv` (per D-16) for Python
environment management. `uv.lock` is the lock file format.

**Convention**: the env lock must include all transitive dependencies
with pinned versions. `uv lock` produces this; `pip freeze` is
insufficient because it does not include hashes.

## The reproducibility folder layout

Every project (R&D or Pure Research) has a `reproducibility/` folder:

```
<project>/reproducibility/
├── data_hashes.txt       # SHA-256 of every data file used
├── shared_pins.txt       # commit hashes of shared/ infrastructure used
├── uv.lock               # python env lock (full file)
├── env_lock_hash.txt     # SHA-256 of uv.lock for quick comparison
└── seed.txt              # random seed(s) used
```

### shared_pins.txt

If the project uses shared infrastructure from `shared/` (per
`references/rd/rd_workflow.md` § Shared infrastructure governance),
each shared module's commit hash is pinned here:

```
# format: <shared module path>  <commit hash>
shared/data_pipeline                a3f7d2e1b9c4f5a2e1d8b7c6a5f4e3d2c1b0a9f8
shared/feature_lib                  b9c4f5a2e1d8b7c6a5f4e3d2c1b0a9f8e7d6c5b4
shared/sanity_check_helpers         c5b4a3f2e1d0c9b8a7f6e5d4c3b2a1f0e9d8c7b6
```

A trial's reproducibility 3-tuple effectively becomes a 4-tuple when
shared infrastructure is involved, since changes to `shared/` can
affect results without the project's own commit hash changing.

### seed.txt

Random seeds used by any stochastic component:

```
# format: <component>  <seed>
numpy_global    42
torch_global    42
sklearn_random_state  42
bootstrap_block_sampler  100
```

For DL / RL / any algorithm with stochastic training: report the
**mean** result over multiple seeds (≥ 3) plus the per-seed values.
Single-seed results are not reproducibility-passing for stochastic
algorithms.

## scripts/reproducibility_stamp.py specification

The helper script captures the 3-tuple at trial time:

```bash
python scripts/reproducibility_stamp.py \
  --project-dir <project_dir> \
  --trial-id <trial_id> \
  --data-paths data/raw/spx_5min_2010_2024.parquet data/raw/vix_daily_1990_2024.parquet \
  [--seed 42]
```

**Outputs**:
1. Updates `<project>/reproducibility/data_hashes.txt` with hashes of
   the named data files
2. Captures `git rev-parse HEAD` in the emitted JSON stamp record
3. Captures hash of `<project>/reproducibility/uv.lock` and writes to
   `env_lock_hash.txt`
4. If `--seed` is provided, appends to `seed.txt`
5. prints a JSON stamp record containing trial ID, timestamp, git commit,
   env lock hash, data hashes, seed, and project directory

The trial notebook or caller persists this record to `results.parquet`, the
trial analysis section, or another durable run log. `shared_pins.txt` is
manually maintained when shared infrastructure is used; the helper does not
auto-detect imports.

**Exit codes**:
- 0: 3-tuple captured, working tree clean, all data files exist
- 1: working tree dirty (uncommitted changes), trial cannot be marked
  reproducible
- 2: data file missing (path resolution failed)
- 3: uv.lock missing or stale (run `uv lock` first)

The promotion gate (`rd_promotion_gate.md` § H,
`pr_promotion_gate.md` § G) requires this script to have been run
with exit 0 for every promotion-eligible or claim-cited trial.

## Verification (re-running a stamped trial)

To verify a trial reproduces:

```bash
python scripts/reproducibility_verify.py --trial-id <trial_id>
```

This script:
1. Reads the trial's recorded 3-tuple from the durable run log
   (`results.parquet`, trial analysis section, or equivalent)
2. Verifies the current Git commit matches (or warns if checked out
   to a different commit)
3. Hashes the current `data/raw/` files and compares to recorded
   hashes (warns on mismatch)
4. Hashes the current `uv.lock` and compares (warns on mismatch)
5. Reports any drift

Drift in any of the 3-tuple components means the trial is no longer
exactly reproducible. The trial's claim does not retroactively become
invalid (the original 3-tuple still uniquely identifies the original
trial environment), but the current environment cannot reproduce it
without checking out the recorded state.

## Common failure modes

| Failure | Symptom | Fix |
|---|---|---|
| Forgot to stamp | Promotion claim made without 3-tuple recorded | Block promotion until stamped |
| Working tree dirty at stamp time | `reproducibility_stamp.py` exit 1 | Commit changes; re-stamp |
| Data file silently changed | Vendor updated data; hash mismatch on re-verify | File deviation; recompute downstream as needed |
| Stale uv.lock | Code requires new package not in lock | `uv lock` to regenerate; commit; re-stamp |
| pip freeze instead of uv lock | Missing transitive pinning | Switch to uv per D-16 |
| Single-seed stochastic result | DL training claim with one seed | Re-run with ≥ 3 seeds; report mean + per-seed |
| Hardcoded paths in code | Trial works on author's machine, fails elsewhere | Use relative paths from project root; verified by stamp script |

## Cross-project reproducibility

When R&D project consumes Pure Research findings (or vice versa) per
`dependent_on_research` (per
`references/rd/capability_map_schema.md`), the consuming project pins
the source project's specific 3-tuple in `decisions.md`:

```markdown
## YYYY-MM-DD: dependent_on_research stamp for <source project>

Source project: <name>
Cited claim: E<id> at status `supported`
Source trial: trial_<NNN>
Source 3-tuple:
  - Data hash: <hash from source project's data_hashes.txt>
  - Git commit: <hash from source project's persisted stamp record>
  - Env lock hash: <hash from source project's env_lock_hash.txt>
```

This ensures cross-project claims remain anchored even as either
project evolves.

## Why not pickle / dill / saved models?

Saving models (.pkl, .pt, .h5) is **not** a reproducibility
substitute:

- The model's behavior depends on the data + code that trained it
- Loading a saved model and getting the same predictions does NOT
  prove the training process is reproducible
- Saved-model-only reproducibility hides bugs that appear during
  training but produce a stable artifact

Use saved models for **caching** (avoid re-training during
development), but the reproducibility 3-tuple is what proves the
trial can be re-derived from scratch.
