# Result Analysis Workflow Reference

## Role of this reference

This file holds the detailed method guidance for `research-result-analysis`. The skill body tells the agent how to move. This reference tells the agent which analysis method to use, what each method can support, where it stops, and what evidence would separate rival explanations.

Result analysis is not a decision process. It is the explanation process between "the run produced material" and "the parent workflow decides what to do with the result."

## Phase 1 - Material reconstruction

Start from the plan path. Reconstruct local state from artifacts that the plan or plan-adjacent files make discoverable:

- plan sections: Proposition and hypothesis trace, Prior-work grounding, Plan, Actual execution, Planned vs Actual, Result analysis if rerunning
- sibling `hypothesis.md`
- parent proposition files: `proposition.md`, `observations.md`, `analyses.md`, `decisions.md`, `paper.md` when present
- run directories: `run_manifest.json`, `logs/stdout.log`, `logs/stderr.log`, `outputs/`, `tables/`, `figures/`, `intermediate/`
- scripts, configs, notebooks, or generated diagnostics referenced by the plan or manifest

Treat untraceable summaries as explanation-scope gaps. They may help phrase a question, but they are not material for the explanation.

For pressure tests, dry runs, or design reviews where no plan path exists, state that the plan-path material boundary is unavailable and analyze only the scenario material. Do not imply that the scenario material was reconstructed from local artifacts.

## Phase 2 - Result-feature ledger

Do this before naming causes. A cause candidate that does not attach to a specific result feature is only a story.

| Ledger item | Ask |
|---|---|
| Aggregate movement | What moved overall, by how much, and against what comparator? |
| Slice or condition movement | Which subgroup, regime, input family, scale, horizon, or condition carries the result? |
| Trace or process shape | What happened over time, across training, in a queue, in control flow, or across state transitions? |
| Concentrated cases | Which failures, successes, outliers, or representative examples carry the pattern? |
| Variance source | Do seeds, sampling, initialization, hyperparameters, or environment variation change the result? |
| Measurement behavior | Did aggregation, weighting, thresholding, instrumentation, or evaluator design shape the visible result? |
| Resource shape | Did latency, memory, IO, batching, locks, scheduling, or contention move with the outcome? |
| Missing material | Which absent artifact would change the why-analysis rather than merely improve reporting? |

For prediction and forecasting, add error sign, magnitude, horizon, residual mean, residual autocorrelation, calibration, train/validation/test gap, and tail concentration when available.

If a secondary metric moves, such as latency during an accuracy experiment, keep it as a separate ledger branch until artifacts show it is part of the same result-producing mechanism. A shared run is not enough to force a shared root cause.

## Phase 3 - Explanatory target

The explanatory target is the puzzle, not the status. It should be phrased as a result feature that needs a generative account.

Good targets:

- "Why did the aggregate improve while the auth-parameter slice worsened?"
- "Why did the forecast underpredict high-volatility windows after horizon 5?"
- "Why did p50 latency improve while p99 worsened?"
- "Why did the proof route work only under the boundary condition k = 0?"
- "Why did one seed diverge while neighboring seeds converged?"

Poor targets:

- "Was the hypothesis supported?"
- "Did the result pass the threshold?"
- "Is the method ready?"
- "What should we do next?"

## Phase 4 - Method routing

Choose the smallest set of methods that can explain the target. Add a method only if it can explain a concrete result feature or distinguish a serious rival.

### Apparent-hit and mixed-results workflow

Use when the prediction seems to hit at aggregate level, or one metric improves, but slices, traces, resource behavior, or concentrated failures move differently.

Inspect:

- slice weights and case counts
- which slice carries the aggregate
- whether the planned mechanism is present in the improving slice
- whether a different mechanism produces the failing slice
- whether the metric or evaluator hides the split
- whether the same mechanism explains both improvement and regression

Can support:

- "The apparent hit is generated by slice A and does not explain slice B."
- "The predicted mechanism is plausible only under condition C."
- "The aggregate movement is explained by benchmark composition plus slice-dependent behavior."

Cannot support by itself:

- "The planned mechanism is confirmed."
- "The hypothesis is supported."
- "The regression is irrelevant because the aggregate passed."

Discriminators:

- slice-balanced metric
- per-source or per-condition evaluation
- retrieval-on/off or component-on/off ablation within each slice
- representative failure sampling from the regressing slice

### Missed-prediction workflow

Use when the planned prediction missed, reversed, or only held under narrower conditions.

Inspect:

- error by slice, condition, horizon, sign, magnitude, time trace, and tail cases
- planned result-producing chain versus observed result-producing chain
- wrong pre-run assumption: missing variable, coarse regime, hidden subgroup, weighting choice, leakage, single seed, unisolated intervention, invalid comparator, or weak measurement
- whether the miss comes from data-generating shift, representation gap, method limitation, training/process dynamics, evaluator weighting, implementation change, random variation, or interaction

Can support:

- "The miss was generated by this specific mismatch between planned assumptions and observed process."
- "No root cause is identified yet; these branches remain live."

Cannot support by itself:

- counterfactual claims such as "the prediction would have held if X were present" without intervention, ablation, limiting-case check, or equivalent causal structure

Discriminators:

- ablation that isolates the missing factor
- matched slice comparison
- controlled perturbation
- residual or failure-family comparison
- independent repeat under the suspected condition

### Causal-factor tree / RCA

Use when the output risks stopping at a proximate label such as "the metric hid it", "the feature did not generalize", "the cache was stale", or "the parser recursed."

Build:

- problem feature: the exact observed feature being explained
- proximate trigger: nearest observed event or condition
- contributing factors: conditions that shaped the result
- current root-cause candidate: deepest supported factor, control gap, or interaction
- why deeper than the trigger: why the trigger existed, dominated, recurred, or escaped controls
- evidence boundary: where artifacts stop supporting deeper inference

Can support:

- a root-cause candidate when the chain is artifact-supported and rival-aware
- a no-root-cause-yet result when only proximate triggers are supported

Cannot support by itself:

- a single certain root cause in a complex result without rival analysis
- a prevention claim unless the control gap is actually evidenced

Discriminators:

- trace that distinguishes trigger from deeper condition
- barrier/control check
- change comparison
- recurrence across independent cases

### Change analysis

Use when the observed condition differs from the expected, baseline, control, or previous non-failing condition.

Inspect:

- code, config, data split, prompt, model, evaluator, dependency, hardware, scheduling, or procedure differences
- exact timing of the change relative to the result feature
- which changed element can generate which part of the shape
- unchanged elements that constrain the explanation

Can support:

- a causal candidate tied to a specific intervention or drift

Cannot support by itself:

- root-cause depth if it only says "X changed" without explaining why that change generated the result and escaped controls

Discriminators:

- revert/restore comparison
- matched baseline with only one changed factor
- differential trace before and after the change

### Barrier / control analysis

Use when a planned check should have exposed or prevented the result shape.

Inspect:

- missing slice check, ablation, sanity check, comparator, metric, threshold, validation, monitor, invariant, guard, or review
- whether the control existed but measured the wrong thing
- whether the control was bypassed, underweighted, or scoped too narrowly

Can support:

- root-cause candidates framed as control gaps
- explanation of why a proximate trigger escaped the plan

Cannot support by itself:

- a claim that the result would not happen under the missing control unless tested or structurally established

Discriminators:

- run the omitted control
- apply the control retrospectively to the same artifacts
- compare cases where the control would and would not fire

### Mechanism decomposition

Use when the result needs explanation in terms of entities, activities, organization, and conditions.

Inspect:

- entities or components involved
- activity order
- intermediate states
- conditions that enable or block each activity
- where the result becomes likely

Mechanism trace format:

`starting condition -> local process/activity -> intermediate state -> result-producing step -> observed result feature`

Can support:

- "This process would make result feature F expected under condition C."

Cannot support by itself:

- counterfactual or causal certainty when based only on association

Discriminators:

- direct state inspection
- intervention on one activity
- limiting-case check
- trace comparison between successful and failing cases

### Cause-effect trace

Use when the result emerges through state transitions, program behavior, training dynamics, proof steps, parser recovery, queue buildup, or optimization passes.

Inspect:

- first divergence from comparator
- intermediate state differences
- step where the outcome becomes hard to reverse
- later effects that are consequences rather than causes

Can support:

- localization of the result-producing step

Cannot support by itself:

- root-cause depth unless it also explains why the divergent state arose

Discriminators:

- delta debugging
- matched trace with one altered condition
- state diff at the first divergence

### Forecast-error diagnostics

Use when a time-series or forecasting result misses.

Inspect:

- residual mean or bias
- residual autocorrelation
- horizon-specific error
- regime-specific error
- calibration
- error concentration in tails
- train/validation/test differences

Can support:

- "The model leaves structure in the residuals."
- "The miss is horizon-specific or regime-specific."

Cannot support by itself:

- a causal explanation for why that structure exists

Discriminators:

- residual features by regime
- blocked or rolling-origin comparison
- model with the suspected missing signal
- holdout period not used in analysis selection

### Distribution-shift decomposition

Use when degradation or surprise may come from changed data.

Inspect:

- harder familiar examples
- changed feature-outcome relationship
- rare, unseen, or underrepresented examples
- covariate, label, concept, temporal, or population shift
- whether the shift affects input composition, target relationship, evaluator behavior, or all three

Can support:

- localization of degradation to a shift type or regime

Cannot support by itself:

- a full method failure claim when representation or evaluator may be responsible

Discriminators:

- matched old/new examples
- reweighting analysis
- feature-conditional performance
- train-on-old/test-on-new and train-on-new/test-on-new contrast

### Variance-source analysis

Use when the observed result may be carried by seeds, sampling, initialization, hyperparameters, or environment variation.

Inspect:

- number of independent runs
- spread across seeds or samples
- outlier run contribution
- whether the same mechanism appears across repeats
- environment or hardware differences

Can support:

- "This run is not enough to identify the result-generating process."
- "The effect is stable across the inspected variance sources."

Cannot support by itself:

- mechanism claims from a single stochastic trajectory

Discriminators:

- repeat with independent seeds
- bootstrap or resampling
- matched environment rerun
- variance decomposition

### Ablation / contribution analysis

Use when the result is attributed to a component, feature, representation, rule, prompt, retrieval source, optimizer step, or proof assumption.

Inspect:

- component on/off behavior
- partial removal or substitution
- interaction with other components
- whether the same result appears without the component

Can support:

- contribution of a component to a result feature

Cannot support by itself:

- a universal causal story if the ablation changes multiple things at once

Discriminators:

- one-factor ablation
- replacement with a neutral alternative
- factorial ablation for interacting components

### Slice / behavioral analysis

Use when aggregate behavior may hide condition-specific behavior.

Inspect:

- natural subgroups, input families, regimes, scale bands, languages, user types, time blocks, horizon bands, or workload classes
- which slices are underweighted or overrepresented
- representative successes and failures within each slice

Can support:

- explanation of an aggregate through slice composition and slice-specific mechanisms

Cannot support by itself:

- mechanism unless paired with trace, ablation, or source-quality evidence

Discriminators:

- slice-balanced metric
- within-slice intervention
- failure sample comparison

### Resource and measurement analysis

Use when latency, throughput, memory, IO, synchronization, batching, queueing, instrumentation, or metric aggregation may create the result.

Inspect:

- p50/p95/p99 and tails, not only mean
- queue length, locks, retries, cache hit rate, batching, memory locality, IO volume, scheduling, and workload size
- instrumentation overhead or measurement bias
- whether hard examples also consume more resources

Can support:

- system-process explanation for performance shape

Cannot support by itself:

- correctness explanation unless resource behavior affects correctness path

Discriminators:

- hold workload constant and vary resource factor
- cap chunk count, batch size, queue depth, or concurrency
- compare same examples under alternate instrumentation

### Representation and evaluator analysis

Use when the chosen variables, abstraction boundary, metric, or evaluator may make the effect appear, disappear, or move.

Inspect:

- missing variables or proxy variables
- label definition
- feature construction
- metric weighting
- threshold sensitivity
- evaluator failure modes
- mismatch between planned construct and measured construct

Can support:

- explanation that the result is generated by representation or evaluator behavior

Cannot support by itself:

- a claim about the underlying phenomenon unless measurement validity is established

Discriminators:

- alternate metric
- manual audit
- construct-validity check
- per-condition evaluator agreement

## Phase 5 - Explanation packets

For each live candidate explanation, write an explanation packet:

- Candidate name
- Chain: `starting condition -> local process/activity -> intermediate state -> result-producing step -> observed result feature`
- Explains: concrete result features
- Does not explain: concrete result features still surprising
- Root-cause role: proximate trigger, contributing factor, root-cause candidate, interaction candidate, or ruled out
- Evidence boundary: where artifacts stop supporting deeper inference
- Competing explanation: nearest serious rival
- Discriminator: smallest analysis that would separate the candidate from the rival

If a candidate explains the aggregate but not a slice, say so. If a candidate explains a slice but not the aggregate, say so. If two candidates are jointly necessary, name the interaction.

## Phase 6 - Root-cause confidence gate

A root-cause statement is allowed only when all six conditions are satisfied:

1. The exact result feature is named.
2. The proximate trigger is named.
3. The analysis explains why that trigger existed, dominated, recurred, or escaped controls.
4. The chain is anchored to artifacts, traces, tables, logs, runs, scripts, or directly inspected material.
5. At least one serious rival is considered, or the material genuinely makes no rival available.
6. A discriminator is named for live uncertainty.

If any condition is missing, write:

`No root cause is identified yet. The leading candidates are <A> and <B>; current artifacts stop at <evidence boundary>; <discriminator> would separate them.`

Do not use "root cause" as a synonym for the nearest observed problem.

## Failure patterns this workflow prevents

| Failure | Corrective motion |
|---|---|
| Treating an apparent hit as confirmation | Route to apparent-hit and mixed-results workflow; inspect slices and whether the predicted mechanism actually generated the aggregate. |
| Treating a missed prediction as "model failed" | Route to missed-prediction workflow; decompose error and compare planned versus observed chains. |
| Calling the nearest factor the root cause | Build the causal-factor tree and apply the root-cause confidence gate. |
| Listing every possible limitation | Keep only candidates tied to concrete result features or discriminators. |
| Using one "5 whys" chain as proof | Build a tree with rivals, interactions, and evidence boundaries. |
| Making counterfactual claims from correlation | Keep them as candidates unless intervention, ablation, limiting-case check, or causal structure supports them. |
| Explaining only failures | Successful aggregates may still require slice, evaluator, or mechanism analysis. |
| Explaining only successes | Missed predictions may reveal real narrower mechanisms; separate the mechanism that worked from the one that failed. |
